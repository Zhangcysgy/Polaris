## 1. 项目概述

### 1.1 一句话定义

**Polaris** 是一个 LLM 驱动的 AI 科研自主发现系统——以大气科学为首发领域，AI 扮演"科研工程师"，人类扮演"所长/守门人"。

### 1.2 核心愿景（E）

> 现有的气象数据太多，但利用率太低。有了 AI 的辅助，我们可以更加高效地利用现有数据，充分挖掘其中的价值，推动大气科学的发展。

Polaris 目的**不是**训练更好的天气预报 AI 模型，而是构建一个 **"AI 科研操作体"**——用 LLM 驱动代码编写、数据分析、结果解读、文献检索、理论验证、方法迁移、迭代修正的全自动闭环。

### 1.3 三大底层逻辑

| 逻辑维度 | 核心痛点 | Polaris 解法 |
|:---|:---|:---|
| **认知层** | LLM 擅长拟合已知，易陷入"数据内卷" | **第一性原理锚点**：所有结论必须接受物理守恒/量纲检验，拒绝纯统计幻觉 |
| **操作层** | AI 快速产出海量"正确但无用"的中间结果 | **惊喜度导航**：优先探索预测残差最大或物理直觉最反常的边缘节点 |
| **组织层** | 单 Agent 易跑偏，多 Agent 易失序 | **五引擎拓扑管控**：方法库→产物追踪→深度验证→全局迁移→自主发现，线性推进 + 持续迭代 |

### 1.4 首发领域

大气科学（沙尘暴起电、遥相关、气候诊断等），以 Sahara 论文（BET-逾渗湿度门控模型）为首发验证案例。

### 1.5 命名

Polaris（北极星）——科研探索中的恒定导航锚点。

## 2. 用户画像与场景

### 2.1 用户画像

**主要用户**：大气科学研究者（当前为张朝阳本人）

| 属性 | 描述 |
|:---|:---|
| **编程能力** | 中高 — 能写 Python 气象分析代码，但不做软件工程 |
| **LLM 使用经验** | 资深 — 已使用 Claude Code CLI / Reasonix / Kimi Code 等多种工具完成多篇论文 |
| **数理基础** | 扎实 — 能推导数学衔接，验证物理自洽性 |
| **领域知识** | 大气科学专业，但对跨领域理论（表面科学、逾渗物理、非线性动力学）需现学 |
| **工作环境** | CLI 为主（Claude Code CLI / Reasonix CLI），Obsidian 做知识管理 |
| **算力资源** | 本地工作站，可访问 DeepSeek API / 智谱 GLM |

### 2.2 核心工作流（现状 → 目标）

| 环节 | 现状（痛点） | Polaris 目标 |
|:---|:---|:---|
| **初稿生成** | AI 写的代码跑不通，反复 debug 数据/坐标/量纲问题 | 方法库提供已验证模板，首次通过率从 ~30% 提升至 >60% |
| **审稿** | 每次审稿手动要求"独立上下文"，但仍被前序对话污染 | 干净房间自动审稿，彻底上下文隔离 |
| **修改** | 多轮修改后"不知道自己改成什么样了"，没有版本锚点 | 反馈主线追踪：每条审稿意见从出生到解决完整轨迹 |
| **收手判断** | 没有客观标准，只能"改不动了"或"精力耗尽"时停 | 系统学习用户质量标准，自动判断"已达到收手标准" |
| **理论验证** | 跨领域理论需全部现学，数学推导+物理直觉验证，更深验证找不到人 | 跨领域专家模拟审查 + 方法论溯源 + 红队模式 |
| **方法迁移** | 同一方法迁移到不同区域需手动改代码、调参数，从未系统做过 | 最小修改优先 + 不匹配升级 + 差异即发现 |
| **创新发现** | 靠"突发奇想"触发（如 Sahara 的理论拼凑），无法系统化复现 | 自主科学发现循环：下载→分析→查文献→补数据→循环 |

### 2.3 用户故事（MVP）

> **研究探索场景**：张朝阳在 Polaris 中输入"分析全球沙尘暴起电的湿度调控机制"。Polaris 从方法库调取 Sahara 验证过的 BET-逾渗框架，自动下载 ERA5 全球数据，将方法迁移到北非、中亚、澳大利亚、北美等沙尘源区。每个区域的迁移结果自动比对，差异标注为"异常"。异常节点自动触发引擎二深度验证——跨领域专家审查+方法论溯源。每周张朝阳在审批面板查看方法库的待确认条目和系统产生的发现简报，裁定方向。系统持续循环——数据→分析→文献→补数据→新发现。

## 3. 产品目标与成功指标

### 3.1 北极星指标（North Star Metric）

**自主产生的、经人类确认有效的"跨领域理论解释"数量**（以 Sahara BET-逾渗模型为基准：1 个）。

### 3.2 代理指标（可每日/每周计算）

| 指标 | 定义 | 基线（现状） | 目标（M3） | 目标（M12） |
|:---|:---|:---|:---|:---|
| **任务完成率** | 用户提出的分析任务中，最终产出有效结果的比例 | ~60%（估算） | 80% | 95% |
| **首次通过率** | Coder 代码不经修改通过 Reviewer 的比例 | ~30%（估算） | 60% | 80% |
| **平均迭代次数** | 每个节点被 Reviewer 打回的平均次数 | 3-5（估算） | 1.5 | 1.2 |
| **上下文污染指数** | 干净房间审稿与同一会话审稿的意见差异度 | —（未量化） | 差异度 > 30% | 差异度 > 50%（说明隔离有效） |
| **用户介入频率** | 人类每做一次"裁定"，系统能自主运行多少个节点 | ~3（估算） | 10 | 20+ |
| **方法库条目数** | 已验证可复用的原子方法数量 | 0 | 30 | 100+ |
| **跨区域差异发现数** | 引擎三发现的"模型在A区域吻合、B区域偏离"的案例数 | 0 | 5 | 30+ |

### 3.3 失败模式监控

| 失败模式 | 检测方式 | 止损策略 |
|:---|:---|:---|
| **方法库污染** | 入库方法在后续使用中失败率 >30% | 自动降级为"候选"，触发重新审查 |
| **审稿严格度衰减** | 连续3次审稿"零意见" | 强制触发红队模式，必须找出至少1个问题 |
| **CRT 节点膨胀** | 每周产生 >500 个节点 | 触发节点合并规则：连续3个同类节点自动折叠 |
| **人类审批积压** | 待确认方法 >20 条超过2周 | 自动降级为"可手动调用但不可自动调用" |

## 4. 五引擎核心功能需求

### 引擎O：方法库

**定位**：Polaris 的"工具箱"——存储、检索、组合标准分析方法，是所有其他引擎的底层依赖。

#### 功能 O-1：三层知识结构

| 层级 | 名称 | 定义 | 示例 |
|:---|:---|:---|:---|
| L1 | **原子方法** | 最小可复用分析单元，独立可调用 | "BET吸附等温线计算"、"二维逾渗电导率标度律"、"RC电荷弛豫" |
| L2 | **编排条目** | 多个原子方法的串联逻辑 + 数据流 | "BET-逾渗湿度门控模型" = L1-BET → L1-逾渗 → L1-RC |
| L3 | **参数实例** | 具体研究中的参数值 + 适用条件 | "Sahara沙尘 BET c=10, h_c=2.0层, σ₀=10⁻⁷ S/m" |

**用户故事**：
- 作为研究者，我可以按"分析目标"检索编排条目，一键应用到新数据
- 作为研究者，我可以按"技术关键词"检索原子方法，自由组合
- 作为研究者，我可以查看任何方法的"使用历史"——哪些研究用过、成功/失败条件

#### 功能 O-2：自生长机制

| 触发条件 | 动作 | 目标层级 |
|:---|:---|:---|
| 分析方法被成功执行并通过引擎二审查 | 自动创建编排条目 + 参数实例 | L1（候选） |
| 某原子方法被 ≥3 个编排条目引用 | 触发"提炼为原子方法"提案，交审批 | L2（独立原子） |
| AI 在分析过程中发现更优方法 | 生成"方法更新提案"，标注新旧对比 | 覆盖原条目 |
| 方法在新场景下失效 | 记录"失效条件"到方法的否定边界字段 | 追加到参数实例 |

#### 功能 O-3：双层质量门禁

```
新方法入库
  ├─ 第一层：引擎二自动审查（方法论溯源+红队模式）
  │   ├─ 通过 → 标记为"待确认"
  │   └─ 不通过 → 标记为"审查未通过"（不可自动调用，但保留以供人工复审）
  └─ 第二层：人类每周审批
      ├─ 确认 → 标记为"已验证"（可自动调用）
      ├─ 驳回 → 标记为"已驳回"（不可调用）
      └─ 超2周未审批 → 自动标记为"候选"（可手动调用，不可自动调用）
```

**用户故事**：
- 作为守门人，我每周花 15-30 分钟审批"待确认"方法，每条看摘要+引擎二审查结论，批量确认/驳回
- 作为守门人，我只审批"方法对我有没有用"——引擎二负责正确性，我负责相关性

#### 功能 O-4：方法检索与调用

- 支持自然语言检索（"如何计算湿度对沙尘起电的抑制效应"）
- 支持结构化检索（按领域、变量、方法类型、验证状态筛选）
- 方法调用自动注入强制校验（`assert` 物理约束、`NaN` 检查等）
- 方法被调用时自动记录：调用时间、数据源、参数、成功/失败状态

### 引擎一：产物追踪与收手

**定位**：Polaris 的"日志系统"——以反馈为主线、版本为副线，全自动捕获，解决"不知道自己改成什么样了"的核心痛点。

#### 功能 1-1：反馈主线追踪

**核心理念**：不追踪"版本"，追踪"意见"。每个审稿意见是一个独立的追踪单元，从出生到解决有完整轨迹。

```
审稿意见 #3："补充X敏感性实验"
  ├─ 创建时间：2026-07-01 14:32
  ├─ 来源：引擎二 · 方法论溯源模式 · 第2轮审稿
  ├─ 状态：已解决
  ├─ 解决历程：
  │   ├─ v5 修改：新增 X 敏感性实验（脚本 scripts/sensitivity_x.py）
  │   ├─ v6 修改：修正 X 的参数范围（0.1→0.5）
  │   └─ v7 Reviewer 复查：通过 ✓
  └─ 最终验证：X 在 [0.1,0.5] 范围内对 η(RH_c) 影响 <3%，结论稳健
```

**用户故事**：
- 作为研究者，我随时可以问"审稿意见第3条改了吗？"，系统精确告诉我是/否、改在哪、改了几次
- 作为研究者，我随时可以看到"当前版本还有哪些意见未解决"
- 作为研究者，我不想手动标记任何东西——系统从 CLI 对话中自动捕获

#### 功能 1-2：自动版本捕获

**触发时机**（无需用户手动操作）：

| 触发事件 | 版本类型 | 自动命名规则 |
|:---|:---|:---|
| 代码运行结束并输出结果（图/数值） | **研究版本** | `[研究] YYYYMMDD-HHMM 分析目标` |
| 收到审稿意见并执行修改 | **审稿迭代版本** | `[审稿-v{N}] 处理意见#{id}` |
| 人类确认"收手" | **里程碑版本** | `[里程碑] YYYYMMDD 研究节点名称` |
| 干净房间审稿完成 | **审稿记录** | `[审稿] YYYYMMDD-HHMM 独立审稿报告` |

**防膨胀策略**：
- 研究版本：同一天同一分析目标的多次运行，自动合并为最新+差异摘要
- 审稿迭代版本：保留完整链，但超过10个版本后自动生成"版本进化摘要"，旧版本压缩归档
- 所有版本存储：元数据（JSON/YAML） + 产物引用（文件路径/哈希），**不复制大文件**

#### 功能 1-3：干净房间审稿

**核心机制**：每次审稿在独立 LLM 会话中运行，完全隔离前序对话上下文。

| 配置项 | 说明 |
|:---|:---|
| **触发方式** | 配置一次：`nature-review` / `nature-review-strict` 默认使用干净房间 |
| **输入** | 仅传入：最新版论文 + 审稿标准（System Prompt） + 可选：指定审稿维度 |
| **输出** | 结构化审稿意见，带元数据（审稿时间、隔离确认、意见ID） |
| **延迟接受度** | 接受 30-60 秒额外延迟换取审稿质量 |
| **污染检测** | 定期对比"干净房间审稿"与"同一会话审稿"的意见差异度，标记异常 |

**用户故事**：
- 作为研究者，我配置一次"nature-review 默认干净房间"，以后每次调用自动隔离
- 作为研究者，我可以在审稿意见中看到"本次审稿在独立上下文中完成"的确认标记

#### 功能 1-4：收手标准自动化

**目标**：系统学习用户的历史"收手"决策，在新任务中自动判断是否已达到收手标准。

| 阶段 | 功能 |
|:---|:---|
| **初期（冷启动）** | 基于规则：连续2轮审稿零意见 + 物理校验全部通过 + 无未解决意见 → 建议收手 |
| **中期（学习）** | 分析用户历史"确认收手"的案例，提取特征（审稿轮数、意见类型分布、修改幅度衰减曲线） |
| **后期（自主）** | 自动判断并提示："已达到您的收手标准（基于过去X个类似项目的决策模式）。建议收手。[Y]确认 [N]继续迭代" |

**用户故事**：
- 作为研究者，我不需要自己判断"改够了没"——系统基于我的历史行为告诉我
- 作为研究者，收手建议是"建议"不是"强制"——我随时可以忽略继续改

### 引擎二：深度理论验证

**定位**：Polaris 的"审查系统"——跨领域专家模拟审查 + 方法论溯源 + 红队模式，守护科学正确性。

#### 功能 2-1：跨领域专家模拟审查

**核心理念**：不预设审查角度。系统自动分析论文/模型，识别其涉及的学科领域，动态生成最相关的专家审查视角。

| 审查维度 | 方法 | 输出 |
|:---|:---|:---|
| **方法论溯源** | 追溯每个关键公式/假设的原始成立条件，逐一核查系统是否满足这些条件 | 每个假设的"成立条件 vs 实际条件"对照表 |
| **红队模式** | 指定"你是持怀疑态度的审稿人，只找致命缺陷" | 3-5 个最尖锐的质疑，必须包含用户可能忽略的角度 |
| **多角色辩论** | 同时模拟 3 个不同领域专家，互相辩论 | 辩论记录 + 一致意见 + 分歧点 |
| **反事实推理** | 问"如果这个模型是错的，哪些观测事实将无法解释？" | 可检验的附加预测清单 |

**用户故事**：
- 作为研究者，我提交一篇论文草稿，系统返回一份"跨领域审查报告"，标注我从没想过的问题
- 作为研究者，我最想要的是发现自己没想到的盲点（排序 D>A>C>B 已验证）
- 作为研究者，我理解"预想不到的角度"无法被预设角色完全覆盖，但系统应尽力拓展我的盲区

#### 功能 2-2：假设敏感性热图

| 步骤 | 说明 |
|:---|:---|
| 1. 自动提取 | 从模型代码/论文中自动提取所有数值假设和参数 |
| 2. 范围设定 | 基于文献自动设定每个参数的合理不确定性范围（±30% 默认，可覆盖） |
| 3. 传播计算 | 单项敏感度：固定其他参数，逐一扫描每个参数 ±30%，计算输出变化率 |
| 4. 可视化 | 生成"阿喀琉斯之踵"热图——哪几个假设对结论影响最大 |

**用户故事**：
- 作为研究者，我知道模型的哪个假设是"命门"——优先验证或优先收集更多证据
- 排序最低但不可缺——不确定时可以跳过，需要时一键生成

#### 功能 2-3：竞争假设生成

**核心理念**：不只验证你的模型，还要主动找"能产生相同结果的其他模型"。

| 步骤 | 说明 |
|:---|:---|
| 1. 问题重述 | 将你的模型要解释的现象，用数学语言重述（不涉及你的具体方案） |
| 2. 文献搜索 | 检索"解释类似现象的其他机制"（跨领域） |
| 3. 模型对比 | 你的模型 vs 竞争模型，在相同数据集上的拟合优度对比 |
| 4. 区分性检验 | 设计一个实验/分析，能区分你的模型和竞争模型（"如果X成立，则我的模型正确；如果Y成立，则竞争模型正确"） |

**用户故事**：
- 作为研究者，我不只想知道"我的模型对不对"，还想知道"有没有更好的解释我没发现"

#### 功能 2-4：文献交叉验证

- 自动检索模型中引用的每个参数值（如 BET c=10）在文献中的实测范围
- 标注：哪些文献支持、哪些文献给出矛盾数据、哪些条件差异可解释差异
- 生成"参数可信度报告"：每个参数的信赖等级（高/中/低/争议）

### 引擎三：全局方法迁移

**定位**：Polaris 的"扩展系统"——将一个区域验证的方法自动适配到全球，发现差异即发现。

#### 功能 3-1：自动方法迁移

**迁移策略：最小修改优先 → 不匹配升级**

```
Level 0：直接复用 — 只改数据源区域，其他不变
  ├─ 运行 → 有结果
  └─ 自动比对引擎检测异常
      ├─ 无显著异常 → 迁移成功，记录"区域兼容"
      └─ 有显著异常 → 升级到 Level 1

Level 1：参数适配 — 自动查找目标区域的本地参数（矿物成分、粒径分布等），替换模型参数
  ├─ 运行 → 有结果
  └─ 自动比对
      ├─ 异常消失 → 迁移成功，记录"参数敏感"
      └─ 异常仍存在 → 升级到 Level 2

Level 2：方程适配 — 目标区域的物理条件可能不满足原模型假设
  ├─ 生成"假设失效诊断报告"
  └─ 标记为"不可迁移区域"，进入人类审查队列
```

**用户故事**：
- 作为研究者，我只提交分析方法 + 源区域代码，系统自动在全球所有可获取数据的区域跑一遍
- 作为研究者，我不需要手动改任何代码——系统从"只换数据"开始，逐步升级适配

#### 功能 3-2：自动比对引擎

| 比对维度 | 方法 |
|:---|:---|:---|
| **模型-模型比对** | 各区域的 η(RH) 曲线形状、拐点位置、渐变率，聚类分析 |
| **模型-观测比对** | 在有验证数据的区域，自动比对迁移结果 vs 本地观测（闪电、沙尘通量等） |
| **差异标记** | 偏离"全球平均行为"超过 2σ 的区域 → 标记为"异常区域" |
| **异常路由** | 异常区域自动进入引擎二的深度验证管线 |

#### 功能 3-3：验证数据智能评估

| 区域验证数据充足度 | 处理策略 |
|:---|:---|
| **充足**（有闪电数据+沙尘数据+野外观测） | 完整迁移+验证，输出"验证通过/失败" |
| **有限**（仅卫星数据） | 降级为"无监督异常检测"，标记显著偏离 |
| **缺失** | 标记为"待验证"，不输出确定性结论 |

**核心理念**：引擎三的价值不在"全球验证"，而在**发现差异**。模型在 Sahara 吻合但在中亚偏离——这不是失败，而是引擎二的燃料。

### 引擎四：自主科学发现循环

**定位**：Polaris 的"大脑"——五引擎全串联的终极形态，半自主运行，关键节点汇报人类。

#### 功能 4-1：自主循环逻辑

```
人类设定大方向："研究全球沙尘起电的湿度调控机制"
  │
  ▼
循环开始（Polaris 自动执行）
  │
  ├─ Step 1：下载数据
  │   引擎四从 ERA5/卫星等自动下载初始数据
  │
  ├─ Step 2：分析方法匹配
  │   查询引擎O方法库，匹配可用的分析方法
  │   ├─ 有匹配 → 调用
  │   └─ 无匹配 → 提案："需要新建分析方法X"，交人类审批
  │
  ├─ Step 3：执行分析
  │   引擎O方法 + 引擎一追踪（自动捕获版本和反馈）
  │
  ├─ Step 4：结果验证
  │   引擎二深度验证（跨领域审查+假设敏感性+竞争假设）
  │
  ├─ Step 5：文献检索
  │   基于结果自动检索相关文献，比对现有知识
  │   发现知识空白 → 生成"值得深挖的方向"
  │
  ├─ Step 6：判断是否需要更多数据
  │   是 → 自动下载新数据（新变量/新区域/新时间范围），返回 Step 3
  │   否 → 进入 Step 7
  │
  ├─ Step 7：区域扩展（可选）
  │   引擎三将分析方法迁移到全球其他区域
  │   发现跨区域差异 → 差异进入 Step 4 深度验证
  │
  └─ Step 8：循环判断
      满足退出条件？→ 生成发现报告，通知人类
      否则 → 返回 Step 3（基于新数据/新方向继续探索）
```

#### 功能 4-2：人类介入节点（关键节点汇报）

| 介入节点 | 触发条件 | 人类决策 |
|:---|:---|:---|
| **需要全新方法** | 方法库无匹配，AI 无法自动组合 | 审批或拒绝新建方法提案 |
| **发现异常信号** | 某区域的模型偏差 > 3σ 或物理围栏触发红色警报 | 裁定：继续深挖 or 标记为"幽灵节点" |
| **需要下载新数据** | 数据不足 AI 无法继续 | 审批数据下载（防无限循环和存储爆炸） |
| **发现"幽灵信号"** | 青藏高原/南极系统性偏差 | 直接发送红色高亮警报，暂停循环 |
| **循环方向漂移** | AI 偏离原定大方向 | 人类裁定：允许新方向 or 拉回原轨道 |
| **每周例行审批** | 固定频率 | 审批方法库待确认条目 + 查看进展简报 |

#### 功能 4-3：退出条件

| 退出类型 | 条件 | 动作 |
|:---|:---|:---|
| **软退出** | 连续 3 次迭代，物理校验残差不再下降 | 生成"收敛报告"，建议收手 |
| **硬退出** | 同一节点被引擎二打回修改超过 5 次 | 冻结节点，标记为"负结果"，Director 裁定 |
| **人类退出** | 人类主动下达"停止"指令 | 生成当前阶段报告 |
| **资源退出** | API 调用量/磁盘/时间超过预设上限 | 暂停并通知人类 |

#### 功能 4-4：发现简报

Polaris 在每次循环节点或每周生成**发现简报**，包含：

1. **本周执行摘要**：循环节点数、新增发现数、死亡节点数
2. **新发现**：哪些结果是首次发现、与现有文献的关系
3. **异常标记**：哪些结果偏离预期、已自动进入深度验证
4. **方法库更新**：本月新增/更新方法数、待审批数
5. **方向建议**：AI 建议的下周探索方向（人类可裁定）

#### 功能 4-5：长程推理与上下文管理策略

**核心挑战**：引擎四需要将五引擎全部串联，持续运行数百步科学探索。LLM 的上下文窗口有限（即使 200K tokens），如果每一步都携带全部历史，步数一多必然溢出。Polaris 采用"拓扑化上下文外部化"策略——不是保留全部历史，而是保留"检索全部历史的能力"。

**设计理念**：

```
不是："每一步都带着全部历史"（单线程 → 溢出）
而是："每一步只带着当前节点的父节点摘要 + 全局状态快照"（拓扑 → 无界）
```

**每个 CRT 节点的上下文数据包**：

当引擎四从第 N 步推进到第 N+1 步时，LLM 收到的是一个精心裁剪的上下文数据包，而非全部历史对话：

```
┌─────────────────────────────────────────┐
│ 上下文数据包（给 LLM）                    │
├─────────────────────────────────────────┤
│ 1. 当前任务：需要做什么                   │
│ 2. 父节点摘要（≤500字）：从哪来的、为什么  │
│ 3. 全局状态快照（≤1000字）：               │
│    - 已确认的发现列表                     │
│    - 当前活跃的假设列表                   │
│    - 最新物理围栏状态                     │
│    - 未解决的异常列表                     │
│ 4. 相关方法（从引擎O检索，按需加载）       │
│ 5. 相关文献摘要（按需检索）               │
│ 6. 人类最新指令                           │
│                                           │
│ ❌ 不包含：前 N-1 步的完整对话历史         │
│ ❌ 不包含：已解决且不相关节点的详情        │
└─────────────────────────────────────────┘
```

**三层压缩金字塔（L1/L2/L3）**：

Polaris 已有的架构天然提供了多个上下文隔离点，这是相对单会话 LLM 的结构性优势：

| 隔离机制 | 对应引擎 | 效果 |
|:---|:---|:---|
| 干净房间审稿 | 引擎一 | 每次审稿从零开始，仅传入论文+审稿标准 |
| 方法库外部化 | 引擎O | 分析方法不占上下文，检索调入、用完释放（类 RAG） |
| 反馈主线追踪 | 引擎一 | 每个意见独立追踪，修改时只需"当前意见+当前版本" |
| 多专家并行审查 | 引擎二 | 每个专家视角独立会话，互不污染，各自产出后汇总 |
| 区域独立迁移 | 引擎三 | 每个区域独立任务，不需知道其他区域结果 |

在这些天然隔离之上，CRT 拓扑增加三层存储：

| 层级 | 保存内容 | 粒度 | 用途 |
|:---|:---|:---|:---|
| **L1 热存储** | 最近 5-10 个节点的完整上下文 | 精细 | 当前探索线程的活跃工作区 |
| **L2 温存储** | 每个节点的结构化摘要（~500字/节点） | 中等 | 长程回顾、异常溯源、方向漂移检测 |
| **L3 冷存储** | 按主题压缩的元节点（合并同类节点）+ 负结果沉积层 + 幽灵信号档案 | 粗粒度 | "此路不通"知识包、全局经验、跨项目知识复用 |

**千步级推理的实际运行模式**：

```
引擎四主循环：
  Step 1-10   → L1（全部在当前上下文或可快速检索）
  Step 11-50  → L1（最近10步）+ L2摘要（前40步）
  Step 51-200 → L1（最近10步）+ L2摘要（前190步）
                + 当需要某历史节点详情时，按需检索 L2
  Step 200+   → L1 + L2 + L3压缩包
                + 自动合并机制：连续3个同类节点折叠为元节点
```

**关键设计约束**：

- **不保留历史，保留检索历史的能力**：每个节点独立存储为结构化记录，可按需通过摘要反向检索完整内容
- **负结果沉积层**：经过验证的"此路不通"结论自动存入 L3，后续探索自动避开，避免重复踩坑
- **幽灵信号档案**：未被解释但反复出现的异常独立存档，跨项目可检索
- **方向漂移检测**：定期比对当前探索方向与原始大方向，偏离时自动触发人类汇报节点

**与求是引擎的对比**：

| | 求是引擎（浙大） | Polaris |
|:---|:---|:---|
| **千步推理实现** | 未公开细节，推测为多Agent协同+状态管理 | CRT拓扑 + L1/L2/L3分层压缩 + 上下文数据包 |
| **长程一致性保障** | 依赖内部Agent协同 | 全局状态快照 + 父节点溯源链 + 方向漂移检测 |
| **失败恢复** | 未提及 | 每个节点独立存储，可从任意节点回退 |
| **负结果利用** | 未提及 | L3冷存储的"负结果沉积层"——已知死路自动避开 |
| **跨会话记忆** | 未提及 | 方法库（引擎O）+ CRT冷存储（L3）提供持久记忆 |

**MVP 目标**：

引擎四首次交付不做千步级——先证明 **50 步闭环**：验证"状态快照 + 父节点追溯 + L1/L2 分层存储"能保证 50 步内不丢失方向、不发生上下文污染。50 步跑通后逐步扩展到百步、千步。

## 5. 非功能需求

### 5.1 性能

| 指标 | 要求 |
|:---|:---|
| **干净房间审稿延迟** | ≤ 60 秒（可接受） |
| **方法库检索响应** | ≤ 5 秒 |
| **引擎三单区域迁移** | ≤ 10 分钟（含数据下载） |
| **引擎四单次完整循环** | ≤ 30 分钟（不含长时间数据下载） |

### 5.2 可用性

- **零手动标注**：用户不需要手动标记版本、方法、反馈——系统从自然操作中自动捕获
- **渐进式复杂度**：新用户只用引擎一，高级用户逐步解锁全部引擎
- **配置一次**：审稿模式、方法库门禁等配置一次，持续生效

### 5.3 上下文隔离

- 干净房间审稿的 LLM 会话与主会话**零共享记忆**
- 每次干净房间审稿仅传入：论文内容 + 审稿标准 + 指定维度
- 定期（每周）自动检测上下文污染指数

### 5.4 数据安全

- 本地优先：分析代码和中间结果优先存储在本地
- 云端 LLM 调用最小化：仅传必要内容，不传完整数据集
- 版本不复制大文件：仅存储元数据 + 文件引用

### 5.5 可扩展性

- 方法库支持插件化——新领域方法可通过标准接口接入
- 引擎二专家审查视角可持续增加（不修改核心逻辑）
- 引擎四循环逻辑可配置（步数、退出条件、汇报频率）

## 6. 不做清单（范围边界）

以下功能**明确不列入 v1.0 开发范围**：

| 不做的功能 | 理由 |
|:---|:---|
| **多用户/团队协作** | 当前仅单人使用，协作功能是过度工程 |
| **Web UI 界面** | 用户明确在 CLI 中工作 |
| **真实图数据库（Neo4j 等）** | CRT 拓扑先降到线性 Markdown 日志 + SQLite，节点超 100 个后再考虑图数据库 |
| **实时数据流监听** | 先跑通"请求-响应"模式，流式监听是后续功能 |
| **自动投稿/发表** | 仅辅助科研，最终决策权在人类 |
| **多模态输入（语音、手写）** | 全部基于文本交互 |
| **非大气科学领域适配** | 首发聚焦大气科学，架构预留扩展性但 v1.0 不实现 |
| **GPU 训练/微调 LLM** | 仅使用 LLM 推理能力，不训练新模型 |

## 7. 风险与假设

### 7.1 前提假设

| 假设 | 验证方式 | 如果假设不成立 |
|:---|:---|:---|
| LLM 能写好气象代码（首次通过率 ≥30%） | 50 题测试基准，测量实际通过率 | 先积累 Skill，不启动 Agent 编排 |
| LLM 跨领域审查能发现人类未察觉的问题 | 选取 3-5 篇已发表论文，对比人工审稿 vs AI 审稿 | 降级为辅助工具，不依赖自主审查 |
| 全球各区域有足够验证数据 | 扫描各沙尘源区可用数据 | 缩减适用区域，仅标记"可验证区" |
| 用户愿意每周花 15-30 分钟审批方法库 | 观察实际审批频率和堆积率 | 降低门禁严格度或增加自动晋升阈值 |

### 7.2 已知风险

| 风险 | 影响 | 缓解策略 |
|:---|:---|:---|
| **Reviewer-Coder 攻击-防御博弈** | LLM 学会写"能通过但无意义"的代码 | 不定期换用不同的物理约束组合审查策略 |
| **方法库污染** | 低质量方法入库导致下游分析系统性偏差 | 双层门禁 + 自动降级 + 失败率监控 |
| **上下文污染残留** | 干净房间隔离不彻底 | 定期对比检测 + 配置验证 |
| **引擎四循环失控** | 无限下载数据/消耗 API | 资源上限 + 人类介入节点 + 退出条件 |
| **"预想不到的角度"永远无法自动化** | 引擎二的审查有天然上限 | 接受上限，多角色辩论+反事实推理最大化覆盖 |
| **物理围栏误杀** | 再分析数据的插值伪影触发硬断言 | 三级分级响应（WARNING/REVIEW/REJECT） |
| **幽灵节点假阳性** | 青藏高原偏差可能只是边界层参数化问题 | 触发后先自动跑多模式对比排除资料系统误差 |

### 7.3 关键开放问题

1. **引擎二 —— "预想不到的角度"如何被 AI 覆盖？** 多角色辩论 + 红队模式 + 反事实推理能覆盖大部分，但"完全意外的角度"本质上是人脑创造性的领域。当前方案：最大化覆盖，接受有上限。
2. **收手标准的冷启动** —— 用户历史数据不足时如何判断？ 初期用规则（连续2轮零意见），积累 >10 个收手决策后切换学习模型。
3. **引擎四的"大方向"如何设定而不跑偏？** 半自主 + 关键节点汇报 + 漂移检测。

### 8.3 竞品分析：求是引擎 vs Polaris

**来源**：《中国科学报》2026-07-16 报道，浙江大学信息与电子工程学院杨怡豪团队发布"求是引擎"——首个具备千步级长程科研推理能力的科学发现系统。

**核心发现**：求是引擎验证了"AI自主科研"的可行性，与 Polaris 方向高度一致，但两者在领域深度、方法论积累和人机关系定义上存在明确差异。

#### 8.3.1 横向对比

| 维度 | 求是引擎（浙大） | Polaris（本系统） |
|:---|:---|:---|
| **发布时间** | 2026-07-16 | PRD 同日完成（独立工作） |
| **核心能力** | 千步级长程科研推理 | 五引擎全串联自主发现循环（千步级目标，50步 MVP 起步） |
| **多Agent架构** | Planner + 方法构建 + 执行 + 分析 + 质疑/风险检查 | 四个角色（合并为两个MVP）+ 五引擎替代角色层 |
| **已验证成果** | 计算物理（数十年难题新解法）、光谱学（新理论框架）、光计算（新机制） | Sahara（BET-逾渗模型，三重验证 + 火星预测被 Nature 证实） |
| **底层理念** | "Loop Engineering"（循环工程） | "物理锚定 · 人机共驾 · 自研进化" |
| **人类角色** | 提出关键问题、判断价值、把握方向、解释结果 | 守门人（每周审批方法库 + 裁定方向 + 关键节点汇报） |
| **验证方式** | 真实光学实验平台，10+小时连续自主运行 | 再分析数据 + 卫星观测 + 文献多线收敛验证 |
| **学科覆盖** | 物理、光学、生命医学、数学等十余个方向 | 首发大气科学，架构预留多学科 |
| **方法知识管理** | 未提及结构化方法库 | 引擎O：三层结构 + 自生长 + 双层门禁 |
| **审稿/审查机制** | 内部"质疑/风险检查"Agent | 引擎二：跨领域专家模拟 + 红队 + 方法论溯源 + 竞争假设 |
| **上下文管理** | 未公开细节 | CRT拓扑 + L1/L2/L3分层压缩 + 上下文数据包 + 干净房间 |
| **区域/跨场景扩展** | 未提及 | 引擎三：三级迁移策略 + 全球自动比对 + 差异即发现 |

#### 8.3.2 Polaris 的差异化优势

| 独有能力 | 说明 | 竞争壁垒 |
|:---|:---|:---|
| **自生长方法库** | 方法不是一次性编写，而是从每次成功分析中自动提取、沉淀、进化 | 积累越久壁垒越高——方法库是 Polaris 的核心知识产权 |
| **反馈主线追踪** | 审稿意见从出生到解决的完整轨迹，保证迭代不跑偏 | 解决了"改了很多轮不知道改成什么样了"的根本痛点 |
| **干净房间审稿** | 每次审稿完全隔离前序上下文，检测并量化污染度 | 求是引擎的"质疑Agent"若在主上下文运行，面临同样的污染风险 |
| **三级物理围栏** | WARNING/REVIEW/REJECT 分级响应，不是一刀切 | 大气科学专属——再分析数据插值伪影不触发硬阻断 |
| **可计算惊喜度** | 不是模糊概念，是 `0.4×残差 + 0.3×异常 + 0.2×物理违反 + 0.1×先验置信度倒数` | 决定算力优先分配给哪些探索方向 |
| **全球方法迁移** | 同一方法在全球不同区域的表现差异就是发现 | 大气科学的独特优势——地球科学天然是多区域对比 |
| **人类守门人模式** | 半自主 + 关键节点汇报 + 每周审批，而非全自动黑箱 | 人机关系更可持续——人类保持控制权但不阻塞自动化 |

#### 8.3.3 战略定位

- **不正面竞争"千步推理"**——求是引擎在物理/光学已有10+小时连续运行验证，Polaris 从50步起步追赶。
- **不比广度比深度**——求是引擎覆盖十余学科，Polaris 首发深耕大气科学一个领域，做透再做广。
- **不比速度比方法论**——Polaris 的核心输出不只是科研成果，更是一套"AI辅助大气科研的可复制方法论"（工具+思想+系统三位一体）。
- **蓝海判断**：**目前没有人在做大气科学领域的 AI 自主科研系统**。求是引擎首发物理/光学，Polaris 首发大气科学——不冲突，互补。
